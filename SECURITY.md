# Security Policy

## Supported Versions

We actively maintain security fixes for the following versions of BrumKit:

| Version | Supported          |
| ------- | ------------------ |
| 0.1.x   | :white_check_mark: |
| < 0.1   | :x:                |

## Reporting a Vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

If you discover a security vulnerability in BrumKit, please disclose it responsibly by emailing us at:

**security@buildinclicks.com**

### What to Include

Please include as much of the following as possible to help us resolve the issue quickly:

- A description of the vulnerability and its potential impact.
- The affected version(s).
- Step-by-step reproduction instructions.
- Any proof-of-concept or exploit code (if applicable).
- Your suggested fix (optional, but appreciated).

### Response SLA

| Action                      | Target                            |
| --------------------------- | --------------------------------- |
| Acknowledgement of report   | Within 48 hours                   |
| Initial assessment & triage | Within 5 business days            |
| Resolution or workaround    | Within 30 days (critical: 7 days) |
| Public disclosure           | Coordinated with reporter         |

We will keep you informed throughout the process. If you do not receive a response within 48 hours, please follow up to ensure your email was received.

### Safe Harbour

We consider security research conducted in good faith to be authorised and will not pursue legal action against researchers who:

- Make a good-faith effort to avoid privacy violations and service disruption.
- Report vulnerabilities promptly and give us reasonable time to resolve them before public disclosure.
- Do not access, modify, or delete data belonging to other users.

## Disclosure Policy

Once a vulnerability has been resolved, we will publish a security advisory on GitHub. We will credit the reporter unless they prefer to remain anonymous.

## Out of Scope

The following are generally considered out of scope:

- Vulnerabilities in third-party dependencies (report to the respective project).
- Issues that require physical access to a user's device.
- Social engineering attacks.
- Denial-of-service attacks that do not demonstrate a meaningful security impact.
